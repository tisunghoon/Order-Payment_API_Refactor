# 부하테스트 결과 정리 — 결제 재고 차감 분산락 정합성 검증

> **무엇을 했나**: 시나리오 D(한정 수량 매진 경쟁)로 Redis 분산락의 정합성을 검증. 300 VU가 재고 1개짜리 상품 10개를 동시에 경쟁 → **정확히 10명만 결제 성공, over-selling 0건**을 2회 연속 재현 확인.
> **실행일**: 2026-06-30 / **프로파일**: `loadtest` (Docker app 컨테이너) / **도구**: k6 v2.0.0
> **관련 문서**: 환경구축 = `부하테스트-환경구축-계획.md`, 구현 = `분산락-구현-결과-정리.md`, 지표 가이드 = `k6-분산락-검증-모니터링-지표-가이드.md`

---

## 1. 검증 질문과 결론

**질문**: "300명이 동시에 재고 1개를 경쟁할 때, 정확히 10명만 결제 성공하고 over-selling이 0건인가? 분산락이 동시성 하에서 깨지지 않는가?"

**결론**: ✅ **통과**. 분산락이 재고 10슬롯을 정확히 직렬화. 2회 연속 동일 결과(결정성 입증).

---

## 2. 테스트 구성 (시나리오 D)

| 항목 | 값 |
|------|-----|
| 대상 상품 | `product_id` 2~11 (10종), 각 재고 1 |
| VU 패턴 | ramping-vus: 0→50(10s) → 300(5s) → 300(30s), gracefulRampDown 5s |
| 요청 흐름 | 상품 조회 → 주문(DIRECT) → 결제 prepare → 결제 confirm |
| 결제 mock | `MockPortOnePaymentProvider` (`MOCK-PAY-{paymentId}` 규약, 실제 금액 DB 조회) |
| 임계값 | `soldout_confirm_success ≤ 10`, `soldout_oversell_detected == 0` |
| 라운드 리셋 | `seed/reset.sql` — 트랜잭션 테이블 TRUNCATE + 재고 1 복원 |

---

## 3. 정합성 검증 결과 (핵심)

### 3-1. k6 임계값 (애플리케이션 레벨)

| 메트릭 | 기대 | 실제 | 판정 |
|--------|------|------|------|
| `soldout_confirm_success` | 10 | **10** | ✅ |
| `soldout_oversell_detected` | 0 | **0** | ✅ |
| `soldout_order_created` (성공 주문) | 10 | **10** | ✅ |
| `soldout_order_failed_soldout` (정상 거절) | ~20,618 | **20,618** | ✅ |
| checks 실패 | 0 | **0 / 20,628** | ✅ |

### 3-2. DB 사후 검증 (데이터 레벨 — 최종 진실)

```sql
-- 1) PAID 주문 수            → 10   (기대 10) ✅
-- 2) over-selling 행(동일 상품 PAID 2건↑) → 0   (기대 0) ✅
-- 3) 대상 상품 재고 합        → 0   (10종 전부 0) ✅
```

k6(요청 성공)와 DB(실제 커밋)가 **양쪽 모두 10건으로 일치** → 결제 성공과 재고 차감이 원자적으로 묶여 동작함을 이중 확인.

## 4. 재현성 — 2라운드 비교

| 지표 | Round A | Round B | 해석 |
|------|---------|---------|------|
| confirm success | **10** | **10** | 동일 — 결정적 결과 (우연 아님) |
| oversell | 0 | 0 | 동일 |
| iterations | 20,180 | 20,628 | +2% (정상 변동) |
| TPS (http_reqs) | 857/s | 912/s | +6% |
| p90 | 2.04s | 1.83s | -10% |
| p95 | 2.13s | 1.93s | -9% |

**두 번 연속 정확히 10건 성공** = 분산락이 동시성 하에서 결정적으로 동작한다는 강력한 증거. 성능 소폭 개선은 워밍업/캐시 효과 범위 내 정상 변동.

## 5. 성능 지표 (Round 0 베이스라인)

| 지표 | 값 (Round B 기준) |
|------|------|
| 처리량 | 912 req/s, 456 iter/s |
| http_req_duration p50 | 27.5ms |
| http_req_duration p90 / p95 | 1.83s / 1.93s |
| iteration_duration p95 | 2.04s |
| http_req_failed | 49.95% (= 품절 정상 거절, **시스템 오류 아님**) |

> **`http_req_failed 49.95%` 주의**: 41,276 요청 중 20,618건이 `PRODUCT_SOLD_OUT` 정상 거절. 재고 1개에 30명 경쟁 시 29명 거절이 정확한 동작. 5xx 오류율이 아님.
>
> **p90/p95 ≈ 1.8~2.1s**: 300 VU가 동시에 분산락 대기열에 몰릴 때의 락 경합 지연. 정합성엔 영향 없으며, 이후 라운드(인덱스/캐싱)에서 이 값과 비교할 기준선.

## 6. 부하 테스트 중 발견·수정한 결함

테스트 환경 구축·검증 과정에서 드러난 숨은 결함들(전부 수정·커밋 완료).

| # | 결함 | 증상 | 수정 |
|---|------|------|------|
| 1 | k6 confirm의 `pgTransactionId`가 mock 규약 불일치 | 금액 13000 fallback → 결제 전량 실패 | `MOCK-PAY-{paymentId}`로 수정 |
| 2 | k6 prepare에 `paymentMethod` 누락 | 400(must not be null) → prepare 전량 실패 | `paymentMethod: 'CARD'` 추가 |
| 3 | `product_id` 매핑이 실재 범위(2~11)와 불일치 | 존재하지 않는 상품 조회 | `pickProductIdByVu`·`reset.sql` 범위 보정 |
| 4 | `kakao.redirect-uri` 최상위 키 누락 | 앱 기동 실패(PlaceholderResolution) | `application-loadtest.yml`에 추가 |
| 5 | Docker app이 `local` 프로파일로 기동 | Mock 비활성 → 실제 PortOne 호출 → 502 | `docker-compose.yml` 프로파일을 `loadtest`로 |
| 6 | loadtest의 `application` 태그가 `myfave-api-loadtest` | 기존 Grafana 대시보드 쿼리(`myfave-api`)와 불일치 → 빈 패널 | 태그를 `myfave-api`로 통일 |
| 7 | `MockPortOnePaymentProvider`가 PortOne 메트릭 미기록 | `PortOne PG Call Duration` 패널 영구 공백 | 실제 Provider와 동일한 `invokeWithMetrics` 추가 → **반영 확인(series=1, getPaymentInfo success=10)** |
| 8 | 안전 카운터 패널이 `0` 대신 `No data` | lazy 등록(정상이라 이벤트 0건) | 대시보드 4종에 `or vector(0)` |

## 7. 미해결 / 후속 과제

- **Lock Wait Duration p95 패널**: 메트릭은 수집되나 테스트 종료 후 5m rate 윈도우가 비어 즉시 조회 시 `NaN`. 테스트 중/직후에만 유효. 라이브 관측용으로는 정상.
- **k6 메트릭(VUs/Rate/Duration) 패널**: `k6 run --out experimental-prometheus-rw ...` 플래그로 실행해야 Prometheus에 수집됨. 이번엔 플래그 없이 실행해 공백.
- **시나리오 A(스파이크)·E(채팅 1000 WS)**: 미실행. Chat Broadcast Latency 패널은 시나리오 E 실행 시 채워짐.
- **성능 절대 수치 측정**: 부하 생성기를 별도 머신으로 분리한 정식 성능 라운드는 본 정합성 검증 범위 밖(후속 단계).
- **Round 1~3 비교**: 인덱스 적용/N+1 제거/캐싱 후 본 베이스라인(p95 ≈ 1.9s)과 비교 예정.
