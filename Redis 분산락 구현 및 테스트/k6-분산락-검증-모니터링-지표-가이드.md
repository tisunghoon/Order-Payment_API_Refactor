# k6 분산락 검증 — 모니터링 지표 가이드

> **목적**: 결제 재고 차감을 비관적 락(`@Lock(PESSIMISTIC_WRITE)`) → Redis 분산락(Redisson)으로 전환한 뒤, k6 부하 테스트(**시나리오 D — 매진 경쟁**)로 "분산락이 제대로 동작하는지"를 **어떤 지표로, 무엇이 정상인지 판단하며** 검증하는 실전 가이드.
> **관련 문서**: `결제-동시성-Redis-분산락-전환-계획.md`, `분산락-전환-실행계획.md`

---

## 0. TL;DR — 분산락이 "잘 됐다"의 정의 3가지

분산락 검증은 결국 아래 3개 질문에 "예"라고 답하는 것이다. 각 질문마다 봐야 할 핵심 지표를 묶어 둔다.

| # | 질문 | 핵심 지표 | PASS 기준 |
|---|---|---|---|
| ① | **Over-selling이 0건인가?** (정합성) | `myfave_stock_snapshot`, `myfave_stock_deduct_attempt_total{outcome="success"}`, DB SQL | 재고 1×10개 → 성공 차감 정확히 **10**, snapshot **0**, 중복 PAID **0행** |
| ② | **DB 커넥션을 점유한 채 대기하지 않는가?** (분산락의 진짜 이점) | `hikaricp_connections_pending`, `hikaricp_connections_active` | 비관적 락 대비 **pending p99가 현저히 낮음** (대기는 Redis가 흡수) |
| ③ | **락 경합이 정상적으로 직렬화되는가?** (안정성) | `myfave_stock_lock_acquire_total{outcome}`, `myfave_stock_lock_wait_duration_seconds` | `outcome="success"`가 다수, `redis_error`/`interrupted` **0**, `timeout`은 의도된 실패라 발생 가능(=락이 실제로 막고 있다는 증거) |

> **한 줄 요약**: `snapshot==0` + `중복 PAID 0행`(정합성) **그리고** `hikaricp_pending`이 낮음(자원 효율) — 이 둘이 동시에 성립하면 분산락 전환 성공.

## 1. 사전 점검 체크리스트

부하 실행 전 아래가 모두 충족돼야 지표가 제대로 수집된다.

- [ ] **Redis 기동** — `docker-compose up -d` (`myfave-redis`, redis:7-alpine). 분산락이 Redis에 의존하므로 필수.
- [ ] **백엔드 loadtest 프로파일** — `SPRING_PROFILES_ACTIVE=loadtest ./gradlew bootRun`
  - `MockPortOnePaymentProvider`(`@Profile("loadtest") @Primary`)가 활성화돼 **실 PortOne PG 호출 없이** 결제 승인이 진행됨. (실 PG를 때리면 부하·요율 제한으로 테스트 불가)
  - HikariCP 50, Tomcat 400 스레드, Hibernate slow-query(500ms) 로깅이 켜짐.
- [ ] **관측 스택 기동** — `docker-compose -f docker-compose.monitoring.yml up -d` (Prometheus+Grafana+Loki+Tempo+Promtail)
  - Prometheus는 `--web.enable-remote-write-receiver` 켜진 상태(k6 메트릭 수신용).
  - Prometheus가 `/api/v1/actuator/prometheus`를 15s 주기로 스크레이프.
- [ ] **토큰 풀 생성** — `node backend/load-test/seed/token-prepare.mjs` → `k6/data/tokens.json` (1000개)
- [ ] **라운드 초기화** — `psql ... -f backend/load-test/seed/reset.sql` (상품 1~10 재고 1로 리셋, 주문/결제 TRUNCATE)

## 2. 모니터링 지표 카탈로그 (핵심)

> Prometheus 표기 규칙: 코드의 `myfave.stock.lock.acquire` → 메트릭 `myfave_stock_lock_acquire_total`(Counter는 `_total` 접미사). Timer는 `_seconds_bucket`/`_count`/`_sum`. 모든 지표에 `application="myfave-api"` 라벨이 붙는다.

### 2-1. 분산락 1군 — 직접 검증 (가장 중요)

#### `myfave_stock_lock_acquire_total{outcome}` — 락 획득 시도 결과
- **타입**: Counter / **태그**: `outcome = success | timeout | interrupted | redis_error`
- **출처**: `RedissonDistributedLockManager.executeWithLock()` (43·48·54·60줄)
- **PromQL (outcome별 분포, 15m 누적)**:
  ```promql
  sum by (outcome) (increase(myfave_stock_lock_acquire_total[15m]))
  ```
- **읽는 법**:
  | outcome | 의미 | 정상 판단 |
  |---|---|---|
  | `success` | 락 획득 성공 | 다수. 시나리오 D면 ≥10 (성공 차감 수만큼) |
  | `timeout` | `waitTime`(3s) 내 락 미획득 | **발생해도 정상** — 경쟁 VU가 줄서다 실패한 것 = 락이 실제로 직렬화하고 있다는 증거. PG 자동환불로 보상됨 |
  | `interrupted` | 대기 중 스레드 인터럽트 | **0이어야 정상**. >0이면 스레드풀/종료 이슈 |
  | `redis_error` | Redis 장애 | **0이어야 정상**. >0이면 Redis 연결/가용성 문제 (모든 결제가 실패로 빠짐) |

#### `myfave_stock_lock_wait_duration_seconds` — 락 대기 시간
- **타입**: Timer(히스토그램, `application.yml`에서 percentiles-histogram 활성) / **출처**: 동 매니저 55·65줄
- **PromQL (p50/p95/p99)**:
  ```promql
  histogram_quantile(0.50, sum(rate(myfave_stock_lock_wait_duration_seconds_bucket[5m])) by (le))
  histogram_quantile(0.95, sum(rate(myfave_stock_lock_wait_duration_seconds_bucket[5m])) by (le))
  histogram_quantile(0.99, sum(rate(myfave_stock_lock_wait_duration_seconds_bucket[5m])) by (le))
  ```
- **읽는 법**: 경합이 심할수록 대기 시간이 길어진다. **p99가 `waitTime`(3s)에 근접**하면 그만큼 줄이 길다는 뜻 → `timeout` 증가와 함께 본다. **중요**: 이 대기는 **DB 커넥션을 잡지 않은 상태**의 Redis 대기다(아래 3군과 대조).

#### `myfave_stock_deduct_attempt_total{outcome}` — 실제 재고 차감 시도
- **타입**: Counter / **태그**: `outcome = success | sold_out | insufficient`
- **출처**: `PaymentService.decreaseStockForConfirm()` (296·299줄) — **락 내부**에서 실행
- **PromQL**:
  ```promql
  sum by (outcome) (increase(myfave_stock_deduct_attempt_total[15m]))
  ```
- **읽는 법**: 시나리오 D에서 `success`는 정확히 **10**(상품 10개×재고 1). 나머지 경쟁자는 락을 얻어 들어왔으나 이미 재고 0 → `sold_out`. `success`가 11 이상이면 **over-selling 버그**.

#### `myfave_stock_snapshot` — 시드 상품 재고 합계 (over-selling 게이지)
- **타입**: Gauge / **출처**: `MetricsConfig` (`sumLoadtestStock`, productId 1~10 합산)
- **PromQL**: `myfave_stock_snapshot`
- **읽는 법**: 시작값 **10**(1×10) → 부하 후 **0**으로 수렴해야 정상. 음수가 될 수는 없고, `시작값 - snapshot == success 차감 수`가 성립해야 정합. **단일 지표로 over-selling을 한눈에** 보는 핵심 게이지.

### 2-2. 결제 플로우 2군 — Saga/보상 검증

| 메트릭 | 타입/태그 | PromQL | 정상 |
|---|---|---|---|
| `myfave_payment_confirm_duration_seconds` | Timer, `outcome` | `histogram_quantile(0.99, sum(rate(myfave_payment_confirm_duration_seconds_bucket[5m])) by (le))` | p99 목표 SLO 이내(락·재고차감 포함 전체 시간) |
| `myfave_payment_confirm_total{outcome}` | Counter, `outcome=success\|failure` | `sum by (outcome) (increase(myfave_payment_confirm_total[15m]))` | success=10, 나머지 failure는 정상(매진/락실패) |
| `myfave_payment_stock_deduct_failed_total{reason}` | Counter, `reason` | `sum by (reason) (increase(myfave_payment_stock_deduct_failed_total[15m]))` | `reason`에 `PRODUCT_SOLD_OUT`/`STOCK_LOCK_ACQUIRE_FAILED` 등장 = 보상 경로 정상 작동 |
| `myfave_payment_pg_auto_refund_total{outcome,trigger}` | Counter | `sum by (outcome,trigger) (increase(myfave_payment_pg_auto_refund_total[15m]))` | `outcome="success"` 위주. `failure`면 운영 알람(환불 누락 위험) |
| `myfave_payment_amount_mismatch_total` | Counter | `increase(myfave_payment_amount_mismatch_total[1h])` | **0** (금액 불일치 = 계산/보안 버그) |

> **해석 포인트**: 분산락 `timeout`(409 STOCK_LOCK_ACQUIRE_FAILED)이 발생하면, 이미 PAID 처리된 PG 결제는 **자동환불로 보상**된다. 즉 `myfave_payment_pg_auto_refund_total{trigger="stock_deduct_failed"}`가 락 timeout 수와 함께 증가하는 것은 **설계대로 동작 중**이라는 신호다.

### 2-3. 자원 점유 3군 — 분산락의 "진짜 이점" 입증 (★)

> 분산락 전환의 핵심 동기는 **"락 대기 중 DB 커넥션을 점유하지 않는다"**였다. 이 군의 지표가 비관적 락 대비 개선됐음을 보이는 것이 발표/면접의 하이라이트.

| 메트릭 | PromQL | 읽는 법 |
|---|---|---|
| `hikaricp_connections_pending` | `hikaricp_connections_pending{application="myfave-api"}` | **커넥션을 기다리는 스레드 수**. 비관적 락에서는 락 대기 스레드가 커넥션을 쥔 채 줄서서 **치솟음** → 분산락에서는 대기를 Redis가 흡수하므로 **거의 0에 수렴**해야 함. ← 가장 중요한 대조 지표 |
| `hikaricp_connections_active` | `hikaricp_connections_active{application="myfave-api"}` | 활성 커넥션. 분산락에서는 실제 차감 트랜잭션(짧음)만 커넥션을 잡으므로 풀(50) 대비 여유 |
| `hikaricp_connections_acquire_seconds` | `rate(hikaricp_connections_acquire_seconds_sum[1m]) / rate(hikaricp_connections_acquire_seconds_count[1m])` | 커넥션 획득 대기 평균. 분산락에서 낮게 유지 |
| `tomcat_threads_busy` | `tomcat_threads_busy{application="myfave-api"}` | 바쁜 톰캣 스레드. 분산락 대기는 톰캣 스레드는 점유하되 **DB 커넥션은 안 잡는다**는 점에 유의 |

> **Before/After 비교 시나리오**: 동일 부하(시나리오 D)를 (a) 비관적 락 버전, (b) 분산락 버전으로 각각 돌려 `hikaricp_connections_pending`의 peak/p99를 비교. (b)가 낮으면 전환 목적 달성.

### 2-4. k6 측 4군 — 클라이언트 관점

k6 자체 메트릭(stdout 요약 또는 Prometheus remote_write).

| 메트릭 | 의미 | 정상 |
|---|---|---|
| `soldout_confirm_success` (커스텀 Counter) | confirm 200/201 누적 | **≤ 10** (threshold로 강제) |
| `soldout_oversell_detected` (커스텀 Counter) | over-selling 감지 | **== 0** (threshold로 강제) |
| `k6_http_req_duration` (p95) | 클라이언트 체감 지연 | 시나리오별 임계값 이내 |
| `k6_http_req_failed` | 실패율 | 매진 시나리오는 의도적 실패(409/품절)가 많아 단독 판단 금물 — 위 커스텀 카운터로 본다 |
| `k6_http_reqs` | 처리량(TPS) | 라운드 비교용 |

> k6 메트릭을 Grafana에서 보려면 **remote_write로 Prometheus에 전송**해야 한다(§4-2). 안 하면 k6 stdout 요약으로만 확인 가능.

## 3. 합격/불합격 판정표 (시나리오 D 기준)

| 항목 | 지표/방법 | PASS | FAIL 시 의심 |
|---|---|---|---|
| Over-selling 0 | `myfave_stock_snapshot` | `== 0` | 락 미적용/조기 만료(leaseTime < 트랜잭션) |
| 성공 차감 수 | `myfave_stock_deduct_attempt_total{outcome="success"}` | `== 10` | >10이면 이중 차감 |
| 중복 결제 | DB SQL #3 | `0행` | over-selling |
| Redis 정상 | `myfave_stock_lock_acquire_total{outcome="redis_error"}` | `== 0` | Redis 연결/가용성 |
| 인터럽트 없음 | `...{outcome="interrupted"}` | `== 0` | 스레드풀/종료 처리 |
| 락 직렬화 동작 | `...{outcome="timeout"}` | `≥ 0` (발생 정상) | (timeout이 0이고 over-sell이면 락이 안 걸린 것) |
| 커넥션 효율 | `hikaricp_connections_pending` p99 | 낮음(비관적 락 대비↓) | 락을 트랜잭션 안에서 잡았을 가능성 |
| 보상 정상 | `myfave_payment_pg_auto_refund_total{outcome="failure"}` | `== 0` | 환불 누락 — 운영 알람 |

### 4-2. k6 메트릭을 Grafana로 보내기 (라운드 비교 대시보드용)

Prometheus가 remote_write 수신 준비돼 있으므로(`--web.enable-remote-write-receiver`), 아래처럼 실행하면 `k6_http_*` 지표가 Grafana `myfave-loadtest-rounds` 대시보드에 들어온다.

```bash
export BASE_URL=http://localhost:8080/api/v1
export K6_PROMETHEUS_RW_SERVER_URL=http://localhost:9090/api/v1/write
export K6_PROMETHEUS_RW_TREND_AS_NATIVE_HISTOGRAM=true
# 라운드/시나리오 라벨 — k6 메트릭 태그 + 백엔드 로그 MDC에 동시 부착됨
export LOADTEST_SCENARIO=D
export LOADTEST_ROUND=0
export LOADTEST_RUN_ID="$(date -u +%FT%H%M)-D-r0"

k6 run --out experimental-prometheus-rw \
  backend/load-test/k6/scenarios/scenario-d-soldout.js
```

> `LOADTEST_*` 환경변수는 `k6/lib/context.js`의 `loadtestTags()`/`loadtestHeaders()`가 읽어, k6 메트릭 태그(Grafana `$scenario`/`$round` 변수)와 HTTP 헤더(`X-Loadtest-*` → 백엔드 MDC → Loki 로그 라벨)에 동시에 실린다.

### 4-3. 후처리 DB 검증 (시나리오 D)

```sql
-- 1) PAID 카운트 (기대: 10)
SELECT COUNT(*) FROM order_items oi
  JOIN orders o ON oi.order_id = o.order_id
 WHERE o.order_status = 'PAID';

-- 2) 재고 0 확인
SELECT product_id, stock_quantity, is_soldout
  FROM products WHERE product_id BETWEEN 1 AND 10;

-- 3) over-selling 검출 (기대: 0행)
SELECT product_id, COUNT(*)
  FROM order_items oi JOIN orders o ON oi.order_id = o.order_id
 WHERE o.order_status = 'PAID'
 GROUP BY product_id HAVING COUNT(*) > 1;
```

## 6. 해석 가이드 — 비관적 락 vs 분산락

같은 시나리오 D를 두 버전으로 돌렸을 때 기대되는 차이:

| 지표 | 비관적 락 (Before) | 분산락 (After) | 의미 |
|---|---|---|---|
| `hikaricp_connections_pending` p99 | **높음** (락 대기 스레드가 커넥션 점유) | **낮음** (Redis가 대기 흡수) | ★ 전환의 핵심 효과 |
| 락 대기 측정 | `myfave_stock_lock_wait_duration`(DB 대기) | 동명 지표(Redis 대기) | 대기 책임이 DB→Redis로 이동 |
| 실패 코드 | `PessimisticLockingFailureException` | `STOCK_LOCK_ACQUIRE_FAILED`(timeout) | 둘 다 보상 경로로 합류 |
| over-selling | 0 | 0 | **정합성은 동일하게 유지** |

> **스토리라인**: "정합성(over-sell 0)은 그대로 지키면서, DB 커넥션 점유(`hikaricp_pending`)를 줄였다." — 이 한 문장을 두 지표로 증명하는 것이 목표.
