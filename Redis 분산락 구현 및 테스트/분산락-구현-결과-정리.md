# 결제 재고 차감 분산락 — 구현 결과 정리

> **무엇을 했나**: 결제 승인 시 재고 차감 동시성 제어를 DB 비관적 락(`@Lock(PESSIMISTIC_WRITE)`) → **Redis 분산락(Redisson)**으로 전환. 실행계획 수립 → STEP 1~10 코드 작성 → 테스트 보완 → 빌드/환경 이슈 해결 → 커밋까지 완료.

## 1. 왜 전환했나 (한 줄 배경)

재고 1개짜리 단일 상품에 결제 트래픽이 몰리면, 비관적 락은 **같은 row 락을 기다리는 스레드가 DB 커넥션을 점유한 채 대기** → HikariCP 풀(기본 10) 고갈. 직렬화 책임을 DB row 락에서 **Redis 분산락**으로 옮겨, 대기를 Redis(pub/sub)가 흡수하게 했다. **대기 중 DB 커넥션 미점유** + over-selling 방지는 그대로 유지.

**확정 결정 3가지**: ① Redisson 추가 · ② 락 실패 정책 = 짧게 대기 후 실패(waitTime ~3s, 실패 시 PG 자동환불 보상) · ③ DB 안전망 = 분산락만(`@Version` 미도입, 후속 과제).

## 2. 핵심 설계 원칙 — 락 경계는 트랜잭션 "바깥"

이 전환의 정확성을 좌우하는 단 하나의 불변식.

```
[confirmPayment] — @Transactional(propagation = NOT_SUPPORTED), 트랜잭션·DB커넥션 없음
  └─ lockManager.executeWithLock(...)         // Redis 락 대기 (커넥션 점유 X)
       └─ self.decreaseStockForConfirm()       // ← @Transactional 시작 → 재고 차감 → 커밋 완료 후 리턴
  └─ finally: 역순 unlock                       // 커밋 이후 해제 보장
```

- `confirmPayment`는 `@Transactional(NOT_SUPPORTED)` **오케스트레이터**(트랜잭션 없음).
- 재고 차감은 `self.`(=`@Lazy PaymentService self`, 프록시) 경유 호출 → **별도 `@Transactional` 시작**. 프록시 경유 메서드는 **정상 리턴 시점 = 커밋 완료 시점**이므로, 람다 반환 직후의 finally unlock은 항상 커밋 뒤에 일어난다.
- **안티패턴(회피함)**: 락을 `decreaseStockForConfirm` *내부*(트랜잭션 안)에서 잡으면 unlock이 커밋보다 먼저 일어날 수 있고, 락 대기 중 커넥션을 점유해 원래 문제가 재발한다. → 그래서 락은 반드시 오케스트레이터에서.
- **데드락 회피**: 복수 상품은 productId **오름차순 정렬** 후 순차 획득, **역순 해제**.

## 3. 실제 구현 결과 (파일별)

### 3-1. 신규 파일

#### `global/lock/DistributedLockManager.java` — 인터페이스
```java
public interface DistributedLockManager {
    // keys 정렬 → 순차 tryLock → action 실행 → finally 역순 해제. 실패 시 STOCK_LOCK_ACQUIRE_FAILED.
    <T> T executeWithLock(List<String> keys, long waitTimeMs, long leaseTimeMs, Supplier<T> action);

    default void executeWithLock(List<String> keys, long waitTimeMs, long leaseTimeMs, Runnable action) {
        executeWithLock(keys, waitTimeMs, leaseTimeMs, () -> { action.run(); return null; });
    }
}
```
> `Supplier<T>`(반환값 있음 — 취소 응답) + `Runnable`(반환값 없음 — 재고 차감) 두 오버로드.

#### `global/lock/RedissonDistributedLockManager.java` — 구현체
- 키 `distinct().sorted()`로 정렬 → `ArrayDeque`에 push(역순 해제용)
- `lock.tryLock(waitTimeMs, leaseTimeMs, MILLISECONDS)`
- `InterruptedException` → `Thread.currentThread().interrupt()` + `outcome=interrupted` 카운터 + 예외
- `tryLock=false` → `outcome=timeout` 카운터 + 예외
- 전부 획득 시 → `outcome=success` 카운터 + `wait.duration` Timer stop → `action.get()`
- `RedisException` → `outcome=redis_error` 카운터 + 예외
- `finally`: 역순으로 `isHeldByCurrentThread()` 가드 후 `unlock()` (leaseTime 만료로 이미 해제된 경우 WARN만)

#### `global/lock/LockKeys.java` — 키 네이밍 유틸
```java
private static final String PRODUCT_STOCK_PREFIX = "lock:product:stock:";
public static String productStock(Long productId) { return PRODUCT_STOCK_PREFIX + productId; }
public static List<String> productStocks(List<Long> productIds) { ... } // map → toList
```
→ 키 형식 `lock:product:stock:{productId}`

#### `global/config/RedissonConfig.java` — RedissonClient 빈
```java
@Bean(destroyMethod = "shutdown")
public RedissonClient redissonClient() {
    Config config = new Config();
    var single = config.useSingleServer()
            .setAddress("redis://" + host + ":" + port)
            .setConnectionPoolSize(16).setConnectionMinimumIdleSize(4)
            .setTimeout(3000).setRetryAttempts(2);
    if (password != null && !password.isBlank()) single.setPassword(password);
    return Redisson.create(config);
}
```
- `@Value`로 `spring.data.redis.host/port/password(옵셔널)` 재사용 → 기존 Lettuce 기반 `RedisTemplate`/채팅 pub/sub/refresh-token과 **별도 커넥션으로 공존**.

### 3-2. 수정 파일

#### `global/error/ErrorCode.java`
```java
STOCK_LOCK_ACQUIRE_FAILED(409, "재고 처리 락 획득 실패 — 잠시 후 다시 시도해 주세요"),
```

#### `domain/payment/service/PaymentService.java` — 핵심
**필드 주입**:
```java
private final DistributedLockManager lockManager;
@Lazy private final PaymentService self;                       // 프록시 자기호출용
@Value("${myfave.lock.product-stock.wait-ms:3000}")  private long lockWaitMs;
@Value("${myfave.lock.product-stock.lease-ms:10000}") private long lockLeaseMs;
```

**confirmPayment** (`@Transactional(NOT_SUPPORTED)`) — 재고 차감을 락으로 감쌈:
```java
try {
    lockManager.executeWithLock(
            LockKeys.productStocks(ctx.sortedProductIds()), lockWaitMs, lockLeaseMs,
            () -> self.decreaseStockForConfirm(ctx.paymentId()));
} catch (CustomException e) {
    meterRegistry.counter("myfave.payment.stock.deduct.failed", "reason", e.getErrorCode().name()).increment();
    // PG는 이미 PAID → 자동 환불 보상 + failConfirm + 원본 에러 전파
    try { paymentProvider.cancelPayment(...); /* auto.refund success */ }
    catch (Exception ex) { /* auto.refund failure — 운영 알람 */ }
    self.failConfirm(ctx.paymentId(), pgInfo.pgTransactionId(), failReason);
    throw e; // PRODUCT_SOLD_OUT / STOCK_INSUFFICIENT / STOCK_LOCK_ACQUIRE_FAILED 그대로 전파
}
```
> **catch 타입을 `CustomException`으로 유지**한 게 설계 포인트: 락 획득 실패가 **기존 재고차감 실패 보상 경로(PG 자동환불)에 자연 합류**한다. 새 분기 불필요.

**decreaseStockForConfirm** (`@Transactional`) — 비관적 락 제거:
```java
List<Long> sortedProductIds = orderItems.stream()
        .map(item -> item.getProduct().getProductId())
        .sorted(Comparator.naturalOrder()).toList();
for (Long pid : sortedProductIds) {
    Product product = productRepository.findByProductIdAndDeletedAtIsNull(pid)  // findByIdForUpdate 대체
            .orElseThrow(() -> new CustomException(ErrorCode.PRODUCT_NOT_FOUND));
    try { product.decreaseStock(1); /* deduct.attempt outcome=success */ }
    catch (CustomException e) { /* outcome=sold_out|insufficient */ throw e; }
}
```

**cancelPayment** (`@Transactional(NOT_SUPPORTED)`) — 전액 취소(재고 복구)만 락 적용:
```java
if (ctx.fullCancel()) {
    return lockManager.executeWithLock(LockKeys.productStocks(ctx.sortedProductIds()), lockWaitMs, lockLeaseMs,
            () -> self.applyCancelResult(ctx.paymentId(), ctx.cancelAmount(), ctx.fullCancel(), userId));
}
return self.applyCancelResult(...);   // 부분취소는 재고 복구 없음 → 락 불필요
```
> `ConfirmContext`/`CancelContext` record에 `List<Long> sortedProductIds` 추가해 오케스트레이터가 락 키를 알도록 함. 웹훅/reconcile은 재고를 안 건드려 락 불필요.

#### `domain/product/repository/ProductRepository.java`
- `findByIdForUpdate`에 `@Deprecated` 표기 (호출부 제거 완료, 메서드 삭제는 후속 PR).

## 9. 설정값 & 후속 과제

**락 파라미터** (프로퍼티 외부화):
- `myfave.lock.product-stock.wait-ms` 기본 `3000` — 짧게 대기 후 실패(실패 시 PG 자동환불 보상)
- `myfave.lock.product-stock.lease-ms` 기본 `10000` — 재고 차감 트랜잭션 최악시간+마진. **leaseTime < 트랜잭션 시간이면 조기 만료 → over-selling 위험**이므로 충분히 크게.

**후속 과제**:
1. `Product`에 `@Version` 낙관적 락 안전망 (Redis 장애/조기 만료 시 DB 레벨 이중차감 최후 방어) — 스키마 마이그레이션 동반, 별도 PR.
2. `ProductRepository.findByIdForUpdate` 완전 제거.
3. Redis HA(Sentinel/Cluster) — 현재 단일 장애점. 다운 시 모든 차감이 `STOCK_LOCK_ACQUIRE_FAILED`(가용성↓, 정합성은 유지).

## 10. 한 줄 결론

비관적 락의 정합성(over-selling 0)은 그대로 유지하면서, **락 대기를 DB 커넥션 밖(Redis)으로 빼내** HikariCP 풀 고갈을 구조적으로 해소했다. 핵심은 "락은 트랜잭션 바깥, 차감은 `self.` 프록시 트랜잭션 안" 한 줄로 요약된다.
