# DB 수정 시 AI가 준수해야 하는 원칙
> 대상 파일: postgresql.example
> 작성일: 2026-03-28

---

## 1. 네이밍 규칙

- **테이블명:** 소문자 스네이크 케이스, 복수형 (`users`, `products`, `payments`)
- **컬럼명:** 소문자 스네이크 케이스 (`user_id`, `created_at`)
- **PK 컬럼명:** `{테이블명_단수}_id` (`user_id`, `product_id`, `coupon_id`)
- **FK 컬럼명:** 참조 대상의 PK명 그대로 사용 (`user_id`, `orders_id`)
- **ENUM 타입명:** `{의미}_enum` (`order_status_enum`, `payment_method_enum`)
- **PK 제약명:** `pk_{테이블명}` (`pk_users`, `pk_orders`)
- **FK 제약명:** `fk_{테이블명}_{참조테이블명}` (`fk_products_users`)
- **UNIQUE 제약명:** `uq_{테이블명}_{컬럼명}` (`uq_users_email`)
- **CHECK 제약명:** `chk_{테이블명}_{설명}` (`chk_products_price`)
- **인덱스명:** `idx_{테이블명}_{컬럼명}` (`idx_products_user_id`)
- **예약어를 컬럼명으로 사용 금지** (`type`, `order`, `user` 등 사용하지 않음)

---

## 2. 컬럼 타입 규칙

| 용도 | 타입 | 비고 |
|------|------|------|
| PK | `BIGINT GENERATED BY DEFAULT AS IDENTITY NOT NULL` | 모든 PK 동일 |
| 시간 | `TIMESTAMPTZ` | `TIMESTAMP` 사용 금지 |
| 상태/코드 값 | ENUM 타입 | VARCHAR로 상태값 저장 금지 |
| 금액 | `INT` | 원화 기준, DEFAULT 0 |
| 불린 | `BOOLEAN NOT NULL DEFAULT {값}` | NULL 허용 금지 |

---

## 3. 필수 컬럼 규칙

### 모든 테이블에 반드시 포함
```sql
"created_at" TIMESTAMPTZ NOT NULL DEFAULT CURRENT_TIMESTAMP
```

### 수정 가능한 데이터가 있는 테이블에 포함
```sql
"updated_at" TIMESTAMPTZ NOT NULL DEFAULT CURRENT_TIMESTAMP
```

> `updated_at`은 DB가 자동 갱신하지 않음. 백엔드에서 UPDATE 시 직접 세팅해야 함.

### `updated_at`이 없는 테이블 (immutable 데이터)
- `order_items` - 주문 당시 스냅샷
- `short_forms` - 생성 후 수정하지 않음
- `style_feeds` - 생성 후 수정하지 않음

위 테이블에 `updated_at`을 추가하지 말 것.

## 4. 제약조건 규칙

### NOT NULL / NULL
- 새 컬럼 추가 시 비즈니스적으로 필수인 값은 반드시 `NOT NULL`
- `NULL` 허용은 선택적 데이터(배송 요청사항, 상세 주소 등)에만 사용
- BOOLEAN 컬럼은 항상 `NOT NULL DEFAULT {값}`

### DEFAULT
- `created_at`, `updated_at`: `DEFAULT CURRENT_TIMESTAMP`
- BOOLEAN: 반드시 DEFAULT 지정 (`DEFAULT FALSE`, `DEFAULT TRUE`)
- ENUM 상태 컬럼: 초기 상태를 DEFAULT로 지정 (예: `DEFAULT 'PENDING'`)
- 금액 컬럼: `DEFAULT 0`

### UNIQUE
- UNIQUE 제약이 있는 컬럼에 별도 인덱스 생성 금지 (UNIQUE가 이미 인덱스를 생성함)
- 복합 UNIQUE의 선행 컬럼에 대한 단일 인덱스도 중복이므로 생성 금지

### FK
- FK 컬럼에는 반드시 인덱스 추가 (UNIQUE FK 제외)
- 순환 참조 FK는 `DEFERRABLE INITIALLY DEFERRED` 사용

### CHECK
- 금액/가격 컬럼: `>= 0` CHECK 필수
- 날짜 범위 컬럼: 시작 < 종료 CHECK 필수 (예: `sale_end_at > sale_start_at`)
- 순서 컬럼: `>= 0` CHECK

## 5. 파일 구조 규칙

postgresql.example 파일은 반드시 아래 순서로 작성:

```
1. ENUM 타입 생성
2. 테이블 생성
3. COMMENT 추가
4. PK 제약조건
5. UNIQUE 제약조건
6. FK 제약조건
7. CHECK 제약조건
8. 인덱스
9. VIEW
```

순서를 바꾸면 FK 참조 실패 등 실행 오류가 발생할 수 있음.

## 6. 수정 시 체크리스트

### 테이블 추가 시
- [ ] PK: `BIGINT GENERATED BY DEFAULT AS IDENTITY NOT NULL`
- [ ] `created_at TIMESTAMPTZ NOT NULL DEFAULT CURRENT_TIMESTAMP` 포함
- [ ] 수정 가능 데이터면 `updated_at` 포함
- [ ] PK 제약 ALTER TABLE 추가
- [ ] FK가 있으면 FK 제약 + 인덱스 추가
- [ ] COMMENT 추가 (ENUM 컬럼, 특수 목적 컬럼)

### 컬럼 추가 시
- [ ] NOT NULL이면 DEFAULT 값 지정 (기존 데이터 호환)
- [ ] FK 컬럼이면 FK 제약 + 인덱스 추가
- [ ] 금액/가격이면 CHECK >= 0 추가
- [ ] 상태/코드 값이면 ENUM 타입 사용

### ENUM 추가/수정 시
- [ ] ENUM 타입은 파일 최상단(섹션 1)에 작성
- [ ] PostgreSQL에서 ENUM 값 삭제는 직접 지원하지 않음 (값 추가만 가능)
- [ ] COMMENT가 있었다면 ENUM 전환 후 불필요한 COMMENT 제거

### 인덱스 추가 시
- [ ] UNIQUE 제약이 이미 있는 컬럼에 중복 인덱스 생성하지 않음
- [ ] 복합 UNIQUE의 선행 컬럼과 중복되지 않는지 확인

---

## 7. 절대 하지 말 것

- `TIMESTAMP` 사용 (반드시 `TIMESTAMPTZ`)
- `GENERATED ALWAYS AS IDENTITY` 사용 (반드시 `BY DEFAULT`)
- VARCHAR로 상태/코드 값 저장 (ENUM 사용)
- BOOLEAN에 NULL 허용 또는 DEFAULT 누락
- FK 컬럼에 인덱스 누락
- UNIQUE와 동일한 인덱스 중복 생성
- 예약어를 컬럼명으로 사용 (`type`, `order`, `user`, `status` 등은 문맥에 따라 주의)
- 파일 섹션 순서 변경

## 8. 현재 DB의 특수 설계 사항

### orders ↔ payments 순환 참조
```
orders.final_payment_id → payments.payment_id (DEFERRABLE)
payments.orders_id → orders.orders_id
```
- `final_payment_id`는 결제 성공 후 백엔드에서 UPDATE로 세팅
- 트랜잭션 내에서만 동작 (`DEFERRABLE INITIALLY DEFERRED`)

### deliveries ↔ orders 1:1 관계
- `uq_deliveries_orders_id` UNIQUE로 1:1 보장
- 주문당 배송 1건

### live_sale_events VIEW
- `sale_events` 테이블의 `is_live`는 컬럼이 아닌 VIEW로 계산
- 스키마 변경 시 VIEW 재생성 필요 (`CREATE OR REPLACE VIEW`)

### chat_rooms는 판매 이벤트(sale_id) 기준으로 운영
- `chat_rooms.sale_id` → `sale_events.sale_id` FK
- 판매 이벤트 시작 30분 전부터 오픈하는 판매자-고객 소통 채팅방
- 상품(product_id)과 무관하게 이벤트 1개당 채팅방 1개

### chat_messages는 Redis에서 관리
- PostgreSQL에 `chat_messages` 테이블 없음
- `chat_rooms` 테이블만 존재 (방 메타데이터)
