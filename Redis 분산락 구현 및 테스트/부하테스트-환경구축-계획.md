# 부하테스트 환경구축 계획 — 결제 재고 차감 정합성 검증

## 변경 목적

결제 **재고 차감 + 분산락** 기능의 **정합성 검증**을 위한 로컬 부하 테스트 환경을 구축한다.

- 검증 질문: "300명이 동시에 재고 1개를 경쟁할 때 정확히 10명만 결제 성공하고 **over-selling이 0건**인가, 분산락이 동시성 하에서 깨지지 않는가"
- 정합성은 자원 경쟁(observer effect)과 무관하므로 **로컬 단일 머신으로 충분**. 성능 절대 수치(TPS/p95)는 이번 범위에서 제외(부하 생성기 분리가 필요해 별도 단계로 분리).
- 기존 자산 `backend/load-test/k6/scenarios/scenario-d-soldout.js`를 그대로 사용한다(신규 스크립트 미작성).

## 변경 내용

신규 코드 변경 없음. 아래 인프라/실행 환경을 단계적으로 기동한다.

| 단계 | 항목 | 명령 / 동작 | 비고 |
|------|------|-------------|------|
| 1 | Docker daemon | `open -a Docker` | Desktop GUI 기동, daemon ready까지 대기 (현재 미실행) |
| 2 | k6 설치 | `brew install k6` | 현재 미설치, brew는 사용 가능 |
| 3 | 인프라 컨테이너 | `docker-compose up -d postgres redis` | app 서비스 제외, DB/Redis만 |
| 4 | 백엔드 기동 | `cd backend && SPRING_PROFILES_ACTIVE=loadtest ./gradlew bootRun` | 1000 유저·Shipping·SaleEvent 자동 시드 |
| 5 | 헬스 확인 | `curl localhost:8080/actuator/health` | 시드 완료까지 대기 |
| 6 | 토큰 풀 생성 | `node backend/load-test/seed/token-prepare.mjs` | `k6/data/tokens.json` 갱신 (1000 토큰) |
| 7 | 재고 리셋 | `docker exec myfave-postgres psql -U postgres -d myfave -f -` < `seed/reset.sql` | 라운드 시작 전 재고 1로 초기화 (psql 없어 docker exec 경유) |
| 8 | 부하 실행 | `k6 run backend/load-test/k6/scenarios/scenario-d-soldout.js` | VU 300 램프업, over-selling threshold 내장 |
| 9 | 정합성 검증 | README의 SQL 3종 (PAID=10, 재고=0, 동일 product over-sell 0행) | docker exec psql로 실행 |

> 로컬 Docker 컨테이너(`myfave-postgres`, `myfave-redis`)와 로컬 8080 백엔드에만 영향. 외부 서버·DB 무영향. 단, `loadtest` 프로파일이 로컬 PostgreSQL에 시드 데이터를 적재하고 `reset.sql`이 재고를 리셋하므로 **로컬 DB 데이터는 덮어쓰여진다**.

## 주의 사항

- **Docker Desktop은 GUI 앱이라 CLI로 완전 자동 기동 불가.** 1단계는 사용자가 직접 켜거나 `open -a Docker` 실행을 허용해야 함.
- `loadtest` 프로파일 설정 파일(`application-loadtest.yml`) 존재 여부와 시드 동작을 **실행 전 검증**한다(이 계획의 선행 단계).
- 기존 `k6/data/tokens.json`은 토큰 만료 가능성이 있어 6단계로 **재생성**한다.
- `reset.sql`이 로컬 DB의 상품/주문 데이터를 변경하므로, 보존하려는 로컬 데이터가 있으면 미리 백업한다.
- k6와 백엔드가 같은 머신에서 도므로 **TPS·지연 수치는 신뢰하지 않는다**. 이번 산출물은 정합성(over-selling/분산락) 판정에 한정한다.

## 실행 결과 (2026-06-30)

시나리오 D (상품 10개 × 재고 1 × VU 300, 45초) 실행. **정합성 전 항목 통과.**

| 검증 항목 | 기대 | 실측 | 판정 |
|---|---|---|---|
| 결제 성공 수 (k6 `confirm_success`) | 10 | 10 | ✅ |
| PAID order_items | 10 | 10 | ✅ |
| payments COMPLETED | 10 | 10 | ✅ |
| 상품 2~11 재고 | 0·품절 | 전부 0, is_soldout=true | ✅ |
| over-selling (동일상품 PAID 2건↑) | 0행 | 0행 | ✅ |
| 품절 실패 처리 | 나머지 | 91,689건 PRODUCT_SOLD_OUT | ✅ |

> 처리량(TPS/latency) 수치는 부하 생성기(k6)와 대상이 같은 머신이라 참고용. 정합성 판정에는 무관.

### 실행 중 발견·수정한 결함 (커밋)

1. `scenario-d-soldout.js` — confirm의 `pgTransactionId`를 `MOCK-PAY-{paymentId}`로 (mock 금액검증 통과)
2. `http.js` — `pickProductIdByVu` 매핑을 실재 ID `2~11`로 (시드 DB에 product 1 없음)
3. `reset.sql` — 재고 리셋 범위 `2~11`로
4. `application-loadtest.yml` — 누락된 `kakao.redirect-uri` 추가 (기동 실패 해결)
5. `scenario-d-soldout.js` — prepare 요청에 `paymentMethod: 'CARD'` 추가 (필수 필드 누락)
6. `README.md` — 검증 SQL 컬럼명(`orders_id`)·상품 ID 범위(`2~11`) 정정

### 인프라 메모

- 기존 postgres 볼륨의 비밀번호 불일치 → 데이터 보존 위해 `ALTER ROLE postgres PASSWORD 'postgres'`로 정렬.
- JWT accessToken 만료가 짧음 → 토큰 재발급 직후 바로 k6 실행해야 함(발급→reset→run 연속 실행 권장).
- 백엔드 context-path가 `/api/v1`이라 health는 `/api/v1/actuator/health`.
