# Redis 분산락 — 모니터링 검증을 위한 추가 작업 정리

> 결제 재고 차감(오버셀링) 문제를 해결하기 위해 도입한 Redis(Redisson) 분산락이
> **실제로 동작하는지 모니터링으로 검증**하기 위해, 모니터링 레포(`MyFaveMornitoring` / 로컬 `observability/`)에
> 추가·수정해야 할 작업 목록.

---

## 현황 요약 (이미 갖춰진 것)

검증에 필요한 4대 조각은 대부분 존재함.

| 구성 | 상태 | 위치 |
|---|---|---|
| 백엔드 분산락/재고 메트릭 노출 | ✅ | `RedissonDistributedLockManager`, `PaymentService`, `MetricsConfig` |
| Prometheus 스크랩 + Grafana 프로비저닝 | ✅ | `observability/prometheus.yml`, `grafana/provisioning/` |
| 분산락 시나리오 전용 대시보드 (7패널) | ✅ | `grafana/.../myfave-stock.json` |
| 오버셀링 부하테스트 (k6 시나리오 D) | ✅ | `backend/load-test/k6/scenarios/scenario-d-soldout.js` |
| k6 → Prometheus RW 수집 경로 | ✅ | `docker-compose.monitoring.yml` (remote-write 활성) |

**실재하는 분산락 관련 메트릭 (알림/패널에서 바로 사용 가능):**
- `myfave_stock_lock_acquire_total{outcome=success|timeout|interrupted|redis_error}` (Counter)
- `myfave_stock_lock_wait_duration_seconds_bucket` (Timer, 히스토그램)
- `myfave_stock_deduct_attempt_total{outcome=success|sold_out|insufficient}` (Counter)
- `myfave_stock_snapshot` (Gauge, 시드 상품 1~10 재고 합계)

---

## 🔴 발견된 문제: 죽은 알림 (Dead Alert)

`prometheus-alerts.yml`의 `StockDeadlockSpike` 알림이 **존재하지 않는 메트릭** `myfave_stock_deadlock_total`을 참조함.
백엔드 커스텀 메트릭 전체 목록에 `deadlock`이 없음 → **절대 발화하지 않는 알림**.
(Redisson 분산락은 PESSIMISTIC 데드락 대신 `timeout`으로 경합이 드러남)

## 추가 작업 목록 (우선순위순)

### 0순위 — 죽은 알림 교체 (백엔드 수정 불필요)

`prometheus-alerts.yml`의 `StockDeadlockSpike`를 실재 메트릭 기반으로 교체.

```yaml
- alert: StockLockTimeoutSpike
  expr: rate(myfave_stock_lock_acquire_total{outcome="timeout"}[1m]) > 0.5
  for: 2m
  labels: { severity: warning, domain: stock }
  annotations:
    summary: "분산락 획득 timeout 급증"
    description: "동일 상품 동시 결제 경합으로 락 대기 초과. 매진 경쟁 상품 의심."
```

---

### 1순위 — 분산락 전용 알림 신설 (백엔드 수정 불필요)

현재 `myfave.stock` 그룹에 분산락 timeout/wait 관련 알림이 **0개**. 아래 4종 추가.

```yaml
# 락 대기 p99가 lease 한계에 근접 — 경합 심화 신호
- alert: StockLockWaitHigh
  expr: histogram_quantile(0.99, sum(rate(myfave_stock_lock_wait_duration_seconds_bucket[5m])) by (le)) > 2
  for: 3m
  labels: { severity: warning, domain: stock }
  annotations:
    summary: "분산락 대기시간 p99 과다"

# Redis 장애로 락 자체 실패 → 오버셀링 직결 위험
- alert: StockLockRedisError
  expr: increase(myfave_stock_lock_acquire_total{outcome="redis_error"}[5m]) > 0
  for: 1m
  labels: { severity: critical, domain: stock }
  annotations:
    summary: "분산락 Redis 오류 — 오버셀링 위험"

# 불변식 위반: 재고 합계 음수 = 분산락이 동시성을 못 막았다는 직접 증거 (가장 강력)
- alert: StockOversellDetected
  expr: myfave_stock_snapshot < 0
  for: 0m
  labels: { severity: critical, domain: stock }
  annotations:
    summary: "오버셀링 발생 — 분산락 무효화 의심"
```

> `StockOversellDetected`가 **분산락 효과를 증명하는 핵심 신호**. 락이 정상이면 절대 0 미만으로 내려가지 않음.

---

### 2순위 — k6 부하테스트 결과를 대시보드/알림에 연동 (백엔드 수정 불필요)

시나리오 D가 RW로 보내는 메트릭이 Prometheus에 수집되지만 현재 미사용.
`myfave-stock.json` 또는 `myfave-loadtest-rounds.json`에 패널 추가.

- `soldout_oversell_detected` — 기대값 **0**, 1 이상이면 빨강 (Stat 패널)
- `soldout_confirm_success` vs 초기 재고(10) 비교 막대
- (선택) 위 두 지표에 대한 알림: `soldout_oversell_detected > 0` → critical

---

### 3순위 — 락 OFF/ON 대조 대시보드 (k6 라벨 주입 필요)

분산락의 **효과**를 증명하는 가장 설득력 있는 방법 = 같은 부하를 락 끄고/켜고 두 번 돌려 비교.

- `myfave-loadtest-rounds.json`은 이미 round/scenario 라벨 비교 구조 → `lock_enabled=true|false` 라벨 추가
- k6 시나리오에 환경변수로 라벨 주입 (`--tag lock_enabled=false`)
- 비교 패널: 락 OFF → `oversell_detected > 0` & `snapshot < 0`, 락 ON → 모두 정상

## 작업 우선순위 표

| 우선순위 | 작업 | 위치 | 백엔드 선행 |
|---|---|---|---|
| 0 | 죽은 `StockDeadlockSpike` 교체 | `prometheus-alerts.yml` | ❌ |
| 1 | 분산락 timeout/wait/redis_error/oversell 알림 4종 | `prometheus-alerts.yml` | ❌ |
| 2 | k6 오버셀링 지표 패널 | `grafana/.../myfave-stock.json` | ❌ |
| 3 | 락 OFF/ON 대조 대시보드 | `myfave-loadtest-rounds.json` + k6 라벨 | ⚠️ k6 라벨만 |

**0·1·2순위는 백엔드 코드 수정 없이 모니터링 레포 내에서 완결 가능.**

## 검증 실행 절차 (참고)

```bash
docker-compose up -d                                   # app, postgres, redis
docker-compose -f docker-compose.monitoring.yml up -d  # prometheus, grafana
export K6_PROMETHEUS_RW_SERVER_URL=http://localhost:9090/api/v1/write
cd backend/load-test/k6/scenarios
k6 run --out experimental-prometheus-rw scenario-d-soldout.js
```

**기대 결과 (분산락 정상 동작 시):**
- Grafana `myfave-stock`: `stock_snapshot` 10 → 0 으로 수렴 (음수 없음)
- `lock_acquire{outcome="timeout"}` 약 290건 (300 VU 중 10건만 성공)
- `deduct_attempt{outcome="success"}` 정확히 **10**
- k6 threshold: `soldout_confirm_success ≤ 10`, `soldout_oversell_detected == 0` 통과

## (선택) 백엔드 추가 고려 사항

모니터링 레포 작업은 아니지만, 성숙도를 위해 검토할 메트릭:
- **락 내부 순 실행 시간** Timer (락 획득 후 ~ 차감 완료) — 현재는 대기시간만 측정
- **불변식 메트릭**: "락 미획득 상태에서 차감 실행" 감지 (차감 시도 대비 락 성공 비율)
- `myfave_stock_snapshot` Gauge 범위 확대 — 현재 시드 상품 1~10만 추적
